# Business Flow Generator

あらゆるテキスト情報から業務フロー図を自動生成するスキル。
Miroボードにスイムレーン付きフロー図を配置する（メイン）、またはFigJamにMermaidダイアグラムを生成する（フォールバック）。

## 出力先の判定

### メイン: Miro REST API v2
以下の条件を満たす場合にMiroを使う:
- 環境変数 `MIRO_TOKEN` と `MIRO_BOARD_ID` が設定されている
- `curl` が利用可能

### フォールバック: FigJam（Figma MCP generate_diagram）
Miroが使えない場合:
- Figma MCP サーバーが接続済みであれば `generate_diagram` でMermaid → FigJamに配置
- それも使えなければ Mermaid構文をテキスト出力（Mermaid Live Editor等で利用）

**起動時に出力先を判定し、ユーザーに伝える。**

---

## ワークフロー

### Step 1: 環境確認

```bash
# Miro接続テスト
if [ -n "${MIRO_TOKEN}" ] && [ -n "${MIRO_BOARD_ID}" ]; then
  STATUS=$(curl -s -o /dev/null -w "%{http_code}" \
    -H "Authorization: Bearer ${MIRO_TOKEN}" \
    "https://api.miro.com/v2/boards/${MIRO_BOARD_ID}")
  if [ "$STATUS" = "200" ]; then
    echo "✅ Miro接続OK"
  fi
fi
```

Miroが使えない場合はFigJamフォールバックに切り替え、ユーザーに通知する。

### Step 2: 入力テキストの取得と前処理

入力は以下のいずれかの形式で受け取る:

- **テキストファイル** (.txt, .md, .doc): ファイルを読み込む
- **チャットへの直接貼り付け**: ユーザーのメッセージからテキストを取得
- **音声文字起こし**: Whisper、otter.ai等の出力テキスト
- **議事録・ヒアリングメモ**: 会議や打ち合わせの記録
- **Slack/チャットログ**: チャットツールからのエクスポートやコピペ
- **業務マニュアル・SOP**: 既存の手順書
- **要件定義書・PRD**: 仕様ドキュメント
- **メールのやりとり**: 業務プロセスが読み取れるメール
- **箇条書きメモ・口頭説明**: 非構造的テキスト
- **複数ソースの組み合わせ**: 上記を複数まとめて処理

前処理として（該当する場合のみ）:
1. 話者の識別（「A:」「営業部:」など）
2. タイムスタンプの除去（必要に応じて）
3. フィラー・相槌の除去（「えーと」「なるほど」など）
4. 業務に関連する記述の抽出

### Step 3: 業務フローの構造化分析

テキストから以下の要素を抽出・構造化する:

#### 3a. アクター（部署・担当者）→ スイムレーン
- 登場する組織単位・役割を特定
- スイムレーンの数と順序を決定
- 例: 営業部、開発部、経理部、経営企画部

#### 3b. タスク → ノード
- 業務ステップを時系列・論理順で抽出
- 各タスクの担当アクター（どのスイムレーンに配置するか）を特定
- 例: 初回ヒアリング、見積作成、工数算出

#### 3c. 分岐・判断 → 判断ノード
- 条件分岐（承認/却下、Yes/No）を特定
- 分岐先のタスクを紐づけ
- 差戻しループがあれば特定

#### 3d. タイムライン（日時指定がある場合）
- 時系列情報があればタイムラインカラムを設定
- 例: 毎月末日17:00 → 翌月1日→2日 → 翌月4日午前…
- 日時情報がなければタイムラインは省略し、左→右の順序のみ

#### 3e. 使用システム・ドキュメント → システムラベル
- 使用するシステム・ツールを特定（Salesforce、Slack、Excel等）
- どのタスクで使用するかを紐づけ
- ノードの下にラベルとして配置

### Step 4: 中間データの生成と確認

抽出結果を構造化データとしてユーザーに提示し、確認を求める:

```
## 抽出された業務フロー

### スイムレーン（上から順）
1. [アクター名1]
2. [アクター名2]
...

### タイムライン（左から順）※日時情報がある場合のみ
1. [時点1]
2. [時点2]
...

### タスク一覧
| # | タスク名 | 担当 | タイムライン | 種別 | 使用システム |
|---|---------|------|------------|------|------------|
| 1 | 開始 | [アクター1] | [時点1] | 開始 | — |
| 2 | [タスク名] | [アクター1] | [時点1] | タスク | Salesforce |
| 3 | [判断名]? | [アクター2] | [時点2] | 判断 | — |
...

### コネクタ一覧
| From | To | 種別 | ラベル |
|------|-----|------|-------|
| #1 | #2 | 通常 | — |
| #3 | #4 | Yes | Yes |
| #3 | #5 | No/差戻し | No |
...
```

**重要**: この段階でユーザーに「この内容で合っていますか？修正点はありますか？」と確認する。
修正があれば反映してからStep 5に進む。

### Step 5: 出力生成

出力先に応じて分岐する。

---

#### 【Miro出力】 Miro REST API v2でボードに配置

詳細なAPIパターンは `references/miro-api-patterns.md` を参照。

##### 5a. ボードクリア（オプション）
既存アイテムがある場合、ユーザーに確認の上でクリアするか聞く。

##### 5b. 座標計算

**重要: レイアウトの方向は左→右（横方向）。**
理想的なフロー図はスイムレーンが縦に積み上がり、フローが左から右に進む。

レイアウト定数:

```
SWIMLANE_HEIGHT = 250         # スイムレーン高さ（固定）
SWIMLANE_LABEL_WIDTH = 150    # 左端レーン名エリア幅
NODE_WIDTH = 140              # タスクノード幅
NODE_HEIGHT = 60              # タスクノード高さ
DECISION_SIZE = 100           # 判断ノードサイズ
START_END_SIZE = 60           # 開始/終了サイズ
NODE_GAP = 50                 # ノード間の空白（50-60px確保）
TIMELINE_COL_WIDTH = 200      # タイムラインカラム幅
SYSTEM_LABEL_WIDTH = 100      # システムラベル幅
SYSTEM_LABEL_HEIGHT = 30      # システムラベル高さ
SYSTEM_LABEL_OFFSET_Y = 40    # ノード下にラベルを配置する距離
```

##### レイアウト構造（理想形）

```
              | 時点1          | 時点2          | 時点3          |
─────────────┼────────────────┼────────────────┼────────────────┤
 部署A        | [開始]→[タスク1]|                |                |
              |    [Salesforce] |                |                |
─────────────┼────────────────┼────────────────┼────────────────┤
 部署B        |                | [タスク2]→<判断>|                |
              |                |   [Excel]       |                |
─────────────┼────────────────┼────────────────┼────────────────┤
 部署C        |                |                | [タスク3]→[終了]|
─────────────┴────────────────┴────────────────┴────────────────┘
```

**フロー方向**: 左→右が基本、部署をまたぐ時のみ上↓下
**スイムレーン**: 縦に積み上げ（部署ごとに横帯）
**タイムライン**: 上部に横方向のカラムヘッダー（日時情報がある場合のみ）

##### 5c. API呼び出し順序

**必ずこの順序で作成する:**
1. **タイトルテキスト**（フロー図の名称。例: `月次売上報告フロー`）
2. **タイムラインヘッダー**（日時情報がある場合、上部に横並びテキスト）
3. **フレーム（スイムレーン）** — 部署名＋タイムライン情報をタイトルに含める
   - 例: `営業部（毎月末日 17:00まで）`、`経理部（翌月1日〜2日）`
4. **シェイプ（タスクノード、開始/終了、判断、差戻し）** — IDを全て保存
5. **システムラベル** — 各タスクノードの下に配置（IDは不要）
6. **コネクタ** — 保存したノードIDを使って接続 — **必ず最後**

##### 5d. ノード種別と色

| 種別 | shape | fillColor | borderColor | サイズ | 備考 |
|------|-------|-----------|-------------|--------|------|
| 開始 | `circle` | `#D5F5E3` | `#27AE60` | 60×60 | 緑丸。`開始` テキスト |
| 終了 | `circle` | `#D5D8DC` | `#7F8C8D` | 60×60 | グレー丸。`終了` テキスト |
| タスク | `rectangle` | `#FFFFFF` | `#2C3E50` | 140×60 | 白四角。業務ステップ名のみ |
| 判断 | `rhombus` | `#FEF9E7` | `#F39C12` | 100×100 | 黄ひし形。`〇〇？` テキスト |
| 差戻し | `rectangle` | `#FADBD8` | `#E74C3C` | 140×60 | 赤四角。差戻し・修正系 |

**重要**:
- 判断ノードのshapeは `rhombus`（`diamond` ではない。hexagonでもない）
- 判断テキストは必ず `?` で終わる（例: `差異あり？`）

##### 5d-2. システムラベル（ノードとは別オブジェクトとして配置）

使用システム・ドキュメントは **タスクノードの下に別オブジェクト** として配置する。
タスクノード内にテキストとして含めない（例: ❌「売上データ入力(Salesforce)」→ ✅ ノード「売上データ入力」＋ラベル「Salesforce」）。

| 種別 | shape | fillColor | borderColor | サイズ | 配置 |
|------|-------|-----------|-------------|--------|------|
| システム | `round_rectangle` | `#EBF5FB` | `#3498DB` | 100×30 | タスクノードの下40px |
| データソース | `round_rectangle` | `#F5EEF8` | `#8E44AD` | 100×30 | 同上 |
| ドキュメント | `round_rectangle` | `#E8F8F5` | `#1ABC9C` | 100×30 | 同上 |

```
配置イメージ:
  ┌──────────────┐
  │ 売上データ入力 │  ← タスクノード (rectangle, 白)
  └──────────────┘
     ┌─────────┐
     │Salesforce│    ← システムラベル (round_rectangle, 薄青)
     └─────────┘
```

##### 5e. コネクタのルール

- 全コネクタに矢印: `endStrokeCap: "stealth"`
- **同じノードの同じ接続点に複数コネクタを接続しない**
- 複数出力がある場合は異なる接続座標を使う
- Yes/No分岐は **色とキャプションの両方** で明確に区別する
- 差戻しコネクタは **必ず破線** にする（通常フローと視覚的に区別）
- コネクタの形状は `elbowed`（直角折れ線）を基本とする

| 用途 | strokeColor | strokeStyle | caption | 備考 |
|------|-------------|-------------|---------|------|
| 通常フロー | `#2C3E50` | `normal` | — | 黒実線 |
| Yes/承認/差異なし | `#27AE60` | `normal` | `Yes` / `差異なし` 等 | 緑実線 |
| No/却下/差異あり | `#E74C3C` | `normal` | `No` / `差異あり` 等 | 赤実線 |
| 差戻し・ループ | `#E74C3C` | `dashed` | `再突合` / `修正` 等 | 赤破線 |
| データ参照 | `#F39C12` | `dashed` | — | 黄破線 |

**重要**: 分岐の Yes/No は必ずキャプションを付ける。
キャプションがないと判断ノードの分岐先が読み手に伝わらない。

##### 5f. APIレート制限対策

各API呼び出しの間に `sleep 0.1` を挿入してレート制限を回避する。

#### 【FigJamフォールバック】 Mermaid → Figma MCP generate_diagram

Miroが使えない場合はこちらを使用。

Mermaid構文パターンは `references/mermaid-patterns.md` を参照。

##### Mermaid構文ルール
- `flowchart TD` + `subgraph` で部署をグルーピング
- ノードIDは英数字のみ（日本語不可）
- 表示テキストは `[""]` で囲む
- `{}` で判断ノード、`[("")]` でDB表現
- "end" を小文字で使わない

##### FigJam配置
```
Use the Figma MCP generate_diagram tool to create this diagram:
[Mermaid構文]
```

##### テキストのみ出力（最終フォールバック）
MiroもFigma MCPも使えない場合:
- Mermaid構文をテキストとして出力
- Mermaid Live Editor (https://mermaid.live) へのリンクを案内

**Miroの場合:**

## エッジケース対応

### 情報が不足している場合
テキストから業務フローを十分に抽出できない場合は、不明点をリストアップして確認する。
推測で埋めず「この部分はテキストに記載がありませんでした」と明示する。
構造化されたテキスト（マニュアル、PRD等）は情報量が多い傾向にあり、
非構造テキスト（口頭メモ、チャット）は不足が出やすいので特に注意する。

### 複雑すぎるフロー
ステップが20以上になる場合は、フェーズまたは部署ごとに分割して複数の図を生成する。
1つの図に詰め込みすぎない。

### 業務フローに関する情報が少ない場合
入力テキストが業務フローに関する内容でない場合（例: 雑談、技術相談、単純なQ&A）は、
「この内容には業務フローに関する情報が少ないようです」と伝え、
別の可視化方法（マインドマップ、要点まとめ、シーケンス図等）を提案する。
部分的にでもフローが読み取れる場合は、読み取れた範囲でフロー図を生成し、
不明な部分を明示する。

### Miro APIエラー
- 401: トークン期限切れ → 再設定を案内
- 404: ボードIDが不正 → 確認を依頼
- 429: レート制限 → sleep時間を増やしてリトライ

## 前提条件セットアップ

### Miro（メイン）
1. Miroアカウントでアプリを作成 → OAuthトークンを取得
2. 環境変数を設定:
   ```bash
   export MIRO_TOKEN="your-miro-token"
   export MIRO_BOARD_ID="your-board-id"
   ```
3. ボードIDはMiroボードURLの `https://miro.com/app/board/` の後ろの部分

### FigJam（フォールバック）
1. Figma MCPサーバーを接続:
   ```bash
   claude mcp add --transport http figma https://mcp.figma.com/mcp
   ```
2. `generate_diagram` ツールが利用可能であること
